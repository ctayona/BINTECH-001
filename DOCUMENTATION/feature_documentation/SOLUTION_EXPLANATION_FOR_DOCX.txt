Solution Explanation

BinTECH is a smart waste segregation solution that combines a website and ESP32-powered kiosk to improve how waste is disposed of in schools. The website handles user registration, login, QR-based session pairing, rewards tracking, and history viewing, while the hardware automatically detects and sorts waste during each session. Together, they reduce manual sorting errors, prevent contamination of recyclables, and motivate users through a points and rewards system. Admins and maintenance staff can also monitor bins, sessions, logs, and system status in real time, making the whole process more efficient and reliable.

Cloud Deployment

BinTECH was deployed by hosting the Node.js and Express application on Render. The server was configured to use the platform-provided `PORT` value so it could run correctly in the cloud, and the same deployment was used for the user website, dashboard pages, QR endpoints, rewards pages, and admin tools. This allowed the web interface and backend logic to remain in one project instead of being split into separate services.

The cloud deployment centered on connecting the application to the services it depends on. Supabase was used as the main database and storage layer, where the system keeps user accounts, session logs, reward data, bin telemetry, redemption history, and uploaded profile images. Its storage buckets were also used for image-related features, especially for profile pictures and Google-based signups. SendGrid was configured for outbound email services so the system could send password recovery emails, account notifications, OTP messages, and reward-related emails. Google Cloud Console was used to set up Google OAuth, including the client ID and authorized origins needed for Google sign-in and account linking.

To keep the deployment secure and manageable, the application used environment variables instead of hardcoded credentials. These variables connect the deployed app to Supabase, SendGrid, and Google authentication services, while Render provides the public hosting environment that keeps the website accessible online.

System Deployment

The system deployment involved both the cloud backend and the ESP32 hardware working together as one connected setup. First, the Supabase schema was prepared by applying the required migration files so the database could support users, rewards, sessions, collections, telemetry, and other system records. After that, the backend was linked to Supabase using environment variables such as the database URL, anon key, and service role key so the application could read and write data securely.

Once the data layer was ready, the remaining deployment settings were added through environment variables for SendGrid and Google client configuration. This allowed the live system to handle email notifications, password recovery, and Google-based authentication without exposing private keys in the code. The Node.js server was then deployed on Render so the website and API could stay available online for user logins, QR pairing, dashboard viewing, reward redemption, and admin monitoring.

On the hardware side, the ESP32 sorter was flashed with the waste-sorting firmware and physically connected to the components of the machine, including the inductive sensor for metal, capacitive sensor for plastic, IR sensor for paper, ultrasonic sensors for bin verification and occupancy, and servo motors with a motor driver for movement control. The ESP32 was configured to join the available WiFi network so it could communicate with the backend through HTTP requests. During operation, it starts a session when a QR pairing request is received, sends live waste counts and fill-level telemetry to the server, receives backend session updates, and posts bin-full warnings when occupancy limits are reached. This keeps the cloud system and the physical kiosk synchronized so sorting activity, points, emails, and bin status are updated in real time.
