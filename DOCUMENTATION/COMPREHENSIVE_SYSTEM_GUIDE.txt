BinTECH Comprehensive System Guide

1. Project Overview
BinTECH is an intelligent waste segregation and incentive platform. It combines ESP32 hardware, a Node.js and Express backend, Supabase as the database and storage layer, and browser-based user and admin pages. The system detects waste, sorts it, awards points, and provides management tools for administrators.

2. Main Technology Stack
- Backend: Node.js, Express
- Database: Supabase PostgreSQL
- Storage: Supabase Storage
- Frontend: HTML templates with client-side JavaScript and Tailwind CSS CDN
- Hardware: ESP32 with sensors, servo motor, LCD, WiFi, and backend HTTP calls
- Email: SendGrid-based email delivery
- Authentication: Local auth, Google OAuth, and OTP verification flows

6. API Logic Summary
User authentication APIs
- POST /auth/register
  - Existing registration flow
- POST /auth/register/request-otp
  - Sends a signup OTP before account creation
- POST /auth/register/verify-otp
  - Verifies OTP and creates the account only after success
- POST /auth/login
  - User login
- POST /auth/google-login
  - Google login
- POST /auth/forgot-password
  - Sends password recovery OTP
- POST /auth/verify-otp
  - Verifies password recovery OTP
- POST /auth/reset-password
  - Resets password after OTP verification

User and session APIs
- /api/dashboard/...
  - User dashboard statistics, activity, points, and history data
- /api/waste-sorter/api/session/...
  - Session start, update, transfer, and end flows for the ESP32 machine
- /api/rewards/...
  - Rewards, redemption, and transfer operations

Admin APIs
- GET /api/admin/backup
  - Downloads a JSON backup snapshot of the Supabase-backed system tables
- GET /api/admin/backup/restore
  - Redirects to the Supabase dashboard for manual review/restoration context
- POST /api/admin/backup/restore-preview
  - Validates a backup JSON payload and returns a non-destructive restore preview
- GET /api/admin/summary
  - Dashboard summary counts and live stats
- GET /api/admin/analytics/...
  - Analytics and reporting endpoints
- Archive routes under /api/admin/accounts/archive-history/...
  - Snapshot view and safe account restore for archived admin accounts

7. Authentication and OTP Flow
Signup OTP flow
1. User sends signup details to /auth/register/request-otp.
2. Backend validates the fields and checks whether the email is available.
3. Backend generates an OTP and stores a pending signup session.
4. Backend sends the OTP by email.
5. User submits the OTP and signup token to /auth/register/verify-otp.
6. Backend verifies the OTP.
7. Backend creates the user account only after verification succeeds.
8. Backend sends the welcome email after successful creation.

Password recovery OTP flow
1. User requests OTP through /auth/forgot-password.
2. Backend generates and stores a recovery OTP.
3. Backend sends the OTP by email.
4. User verifies OTP with /auth/verify-otp.
5. Backend allows the password reset step.

8. Backup and Restore Behavior
Backup export
- The backup download is a JSON snapshot of important Supabase tables.
- It includes data from admin, user, session, telemetry, bin, reward, collection, route, and schedule tables.
- It also includes metadata like version, timestamps, system info, and warnings.
- This is a manual export for disaster recovery and documentation.

Safe restore behavior
- Restore is intentionally manual and preview-based.
- The restore preview endpoint only validates the backup payload.
- It lists table names, row counts, and missing-table warnings.
- It does not insert, update, or delete live data.
- The separate Supabase redirect button is for opening the project dashboard, not for automated overwrites.

Why Supabase matters for backup
- Supabase is the system of record.
- Backups should reflect Supabase data because that is where live records are stored.
- This makes backup export and restore review traceable and consistent.

9. Hardware Logic Summary
- ESP32 firmware reads the physical sensors and updates classification state.
- A detection window is used so one material can win the cycle.
- The firmware synchronizes with backend session data.
- The LCD shows the active user and current machine state.
- The machine stops or resets when backend state requires it.
- Bin telemetry and verification logic reduce unsafe sorting actions.

10. Security Features
System security
- Authenticated access to user and admin pages
- Admin route protection through middleware
- Role-based access checks for sensitive actions
- Head-role restriction for archive restore
- Server-side secrets and environment variables
- Validation of request payloads and backup payloads
- Audit history for account restoration

Hardware security
- Session lock so only one material wins the current detection window
- Confirmation thresholds to reduce false sensor triggers
- Backend sync so the server remains the source of truth
- Fail-closed behavior when no valid session exists
- Verification logic before sorting action completes

13. Common Panel Questions And Ready Answers
Q: What does the backup contain?
A: It exports a JSON snapshot of the main Supabase-backed tables, plus metadata and warnings.

Q: How is restore made safe?
A: Restore uses a preview-only flow that validates the backup but does not overwrite anything automatically.

Q: Why use Supabase?
A: Supabase is the live data store and storage layer, so it keeps the system consistent and easier to manage.

Q: How does signup OTP work?
A: The system sends an OTP before account creation, verifies the code, then creates the account only after the OTP is correct.

Q: What happens if the hardware sensors misread a material?
A: The detection window and locking logic reduce false classification, and the backend sync helps keep the machine state consistent.

Q: How are points protected?
A: Points are controlled by backend session updates instead of trusting the device alone.

Q: What is the role of the admin dashboard?
A: It centralizes monitoring, analytics, account controls, backups, and operational tools.

Q: What security measures are in place?
A: Authentication, authorization, validation, server-side secrets, restore preview, and audit logging.

14. Short Closing Statement
BinTECH is a connected waste sorting platform that uses hardware automation, Supabase-backed data, and role-based web tools to track waste, award points, and keep backup and restore operations safe.
